# 界面原型迭代提示词记录

## 使用场景

这份提示词用于本项目的界面原型迭代。当前迭代目标是明确需求和项目范围，完成主要静态界面和界面上的主要元素，不提前实现完整后端、真实 AI、数据库、Webhook 或自动修复功能。

## 通用提示词

```text
我们正在做《A8 GitHub仓库问答与Issue分析系统》的界面原型迭代。

请只围绕界面原型迭代工作，不要把技术原型迭代或构造迭代的任务提前做完。

当前迭代需要完成：
1. 调研、分析和定义需求，形成 Vision 文档和 Usecase 模型；
2. 初选语言、工具和框架，并记录学习内容；
3. 设计并实现静态/半静态界面原型；
4. 小组内部评审并改进需求文档和界面原型；
5. 编写迭代评估报告，包含评审记录和开发总结。

界面原型可以使用 React/HTML/原型工具或 Mock 数据实现。重点是让老师和组员看清楚系统功能、页面结构、操作流程和模块边界。

请保持现有项目风格：浅色背景、白色卡片、简单网格布局、清晰标题、少量图标和紧凑信息展示。
```

## 第三部分：项目解析功能提示词

```text
我负责第三部分：项目解析功能（无 AI 部分）。

目标是在界面原型迭代中展示“系统如何帮助开发者理解一个 GitHub 项目结构”，不做真实 AI 分析，也不实现复杂后端解析服务。

请基于已有 GitHub 同步数据或 Mock 数据，设计并实现项目解析展示界面。需要包含：
1. 项目类型或主要技术栈概览；
2. 目录结构或主要目录列表；
3. 文件分类统计，例如源码、测试、文档、配置、依赖、CI/CD；
4. 依赖文件展示，例如 package.json、pyproject.toml、requirements.txt；
5. 测试、文档、配置的概览；
6. 项目架构图或思维导图样式的静态展示。

实现时优先复用现有前端页面结构和样式，不要另起一套 UI 风格。
数据来源优先使用成员1已经同步到前端的仓库快照，例如 files、file_categories、languages、readme。
真实依赖解析、深度代码语义分析、交互式架构图可以放到后续技术原型迭代。
```

## Commit Message 规范

每次提交时，commit message 要直接说明改了什么，避免写成含糊的 `update` 或 `fix`。

推荐格式：

```text
type: short description
```

常用类型：

- `docs:` 文档、提示词、计划、README 修改
- `prototype:` 界面原型或 Mock 数据修改
- `feat:` 功能雏形或可运行能力新增
- `fix:` 修复错误
- `style:` 只改样式，不改逻辑
- `chore:` 配置、依赖、整理类修改

示例：

```text
prototype: add project analysis UI prototype
docs: add UI iteration prompt record
style: align project analysis panel with existing UI
fix: import missing frontend type
```
